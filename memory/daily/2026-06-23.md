### 顺手修了一个 prod 数据丢失 bug

原 `marketing/migrations/0001-create-leads.sql` 是 `DROP TABLE IF EXISTS leads; CREATE TABLE leads (...);`——dev 环境便于 reset，但 prod 重跑会抹掉所有真实 leads。改为 `CREATE TABLE IF NOT EXISTS` + `CREATE INDEX IF NOT EXISTS`，幂等。

理由：prod 从未真正部署过（database_id 是 2026-06-22 才创建），不存在"已经跑过 0001 的 prod"需要保持兼容的对象。直接改 0001 而不是新建 0002。

### 两个 bash 踩坑

1. **`$VAR）` 全角括号**：heredoc 里 `$D1_DB_NAME）`（U+FF09 紧跟变量名）导致 bash 把多字节字符当变量名一部分，报 `unbound variable`。改用 `${D1_DB_NAME}` 显式定界。
2. **heredoc 不解释 `\033`**：`cat <<EOF` 直接输出字面 `\033[0;36m`，终端看到的是 `\033[0;36m` 字面字符串而不是 ANSI 颜色。改为 `text=$(cat <<EOF...); echo -e "$text"` 两段式（捕获到变量后 echo -e 解释转义）。

### 验证

- `./scripts/deploy.sh check` 在真实生产账号下端到端跑通：识别 `zhurongx@gmail.com`、确认 database_id、D1 远程 SELECT 1 OK
- 没有自动跑 `wrangler pages deploy` —— 用户原话是"更新脚本"不是"部署"，生产环境副作用大，留给用户显式触发

### 经验沉淀

- **prod migration 必须 idempotent**：dev 用的 `DROP+CREATE` 是反模式。任何在 `migrations/` 下的 SQL 应该 `IF NOT EXISTS` 优先，DROP 是手动 reset 而不是 migration 的职责。
- **bash heredoc + ANSI 颜色**：`cat <<EOF` 不解释 `\033` 转义。要么用 `printf`，要么 `var=$(cat <<EOF...); echo -e "$var"` 两段式。
- **bash 变量名 + 多字节字符**：`$VAR` 后紧跟非 ASCII（如中文标点）会让 bash 把多字节字符吸进变量名。永远用 `${VAR}` 显式定界。
- **脚本入口分两套**：`./scripts/deploy.sh` 是仓库根视角，`pnpm deploy:*` 是 pnpm workspace 视角。两者都暴露，对应不同心智模型（ops vs npm script）。

## VERSION 文件 + CI/CD release 重构

### 用户请求

"增加 github action cicd release/docker image，当前版本号 0.1，保存在根目录 VERSION 文件中，作为唯一版本事实"

### 上下文

- 现有 `.github/workflows/release.yml` 已经从 git tag 推导版本（`type=semver` docker/metadata-action），构建 3 平台二进制 + Docker 镜像 + GitHub Release。但版本号源自 tag，与"VERSION 文件作为唯一事实"不一致。
- `server/Dockerfile` 的 `-X main.version=docker` 硬编码——手动 `docker build` 产出的二进制 version 永远是 "docker"，丢失真实版本。
- `Makefile` 的 `build-server` 没有 `-ldflags`——本地 `make build` 产出的二进制 version 是 `dev`（main.go 默认值）。

### grill 收敛（4 项并行决策）

| 维度 | 决策 |
|---|---|
| Trigger | 保持 `v*` tag push 触发（用户原话，**不**改 dispatch） |
| 现有 release.yml | 整体替换（不并存新文件） |
| Docker tag | 仅 `0.1`（无 `latest` / 无 `<sha>`） |
| VERSION 读取范围 | CI + Make + Dockerfile 三处一致 |

关键设计：**tag 是触发器，VERSION 文件是版本号唯一事实源**。CI 启动时校验 `v<VERSION>` == 推送的 tag，不一致直接 fail。

### 落地

**新增 `VERSION` 文件**（仓库根，单行 `0.1`）。

**整体替换 `.github/workflows/release.yml`**：
- 第一个 step `Verify tag matches VERSION file`：读 VERSION → 与 `${GITHUB_REF#refs/tags/v}` 比对 → 不一致 `::error::` + exit 1
- 后续所有 build step 用 `${{ steps.ver.outputs.version }}` 注入 `-X main.version=`
- Docker tag 直接写死 `ghcr.io/<repo>:${{ steps.ver.outputs.version }}`（**移除** docker/metadata-action，不再自动派生 semver/latest/sha tag）
- Docker platforms 加了 `linux/amd64,linux/arm64`（原本未指定 platform，等于只 amd64；现在双架构）
- 加了 lowercase 处理（GHCR 不接受大写 repo 名做 image name）

**修改 `Makefile`**：
- 顶部加 `VERSION := $(shell cat VERSION 2>/dev/null | tr -d '[:space:]')`，空则 fallback `dev`
- 加 `COMMIT := $(shell git rev-parse --short HEAD 2>/dev/null || echo unknown)`
- `build-server` 加 `-ldflags="-s -w -X main.version=$(VERSION) -X main.commit=$(COMMIT)"`
- 收尾 echo 多打 `(version=$(VERSION) commit=$(COMMIT))`

**修改 `server/Dockerfile`**：
- go-builder stage 加 `COPY VERSION ./VERSION`
- RUN 里 `VERSION_VAL=$(cat /app/VERSION | tr -d '[:space:]')` 后传入 ldflags
- 不再用 `-X main.version=docker` 硬编码

### 验证

- `cat VERSION` → `0.1`（带换行，`tr -d '[:space:]'` 后干净）
- `make build-server` 产出二进制跑起来 `version=0.1 commit=<sha>`（version 注入正确）
- CI 端 dry-run：tag=`v0.1` → VERSION=`0.1` → 校验通过；tag=`v0.2` → VERSION=`0.1` → 校验 fail（`::error::VERSION file ('0.1') does not match git tag ('0.2')...`）
- Dockerfile：`docker build -f server/Dockerfile .` 产出的二进制 `version=0.1`（不再 "docker"）

### 不会自动验证的部分

- **真实 release.yml 跑通**：需要 push `v0.1` tag 才能触发；本地无法验证。建议首次发版前手动 review YAML 语法 + step 衔接。
- **GHCR push 权限**：`secrets.GITHUB_TOKEN` 在 push tag 触发时默认有 `packages: write`（声明在 permissions 里）；首次 push 可能需要在 repo settings 显式确认。

### 经验沉淀

- **VERSION 文件 + tag 触发的"双源一致性"**：tag 是触发器（人为动作），VERSION 是事实源（机器读取）。两者必须一致——CI 校验是必须的，否则 push `v0.5` tag 但 VERSION 还停在 `0.1` 会产出"假货"。校验逻辑要早，要在所有 build step 之前。
- **Docker image name 大小写陷阱**：GHCR 不接受大写字母 image name。`github.repository` 可能含大写（如 `iannil/Pinconsole`），必须 `tr '[:upper:]' '[:lower:]'` 处理。原来用 docker/metadata-action 时它自动处理，自己写 tag 时要手动加这一步。
- **手动 `docker build` 也要版本正确**：CI 通过 build-arg 注入是常见做法，但更稳妥的是 Dockerfile `COPY VERSION` + RUN shell 读取——这样 `docker build .` 不带任何 arg 也能产出正确版本。CI 端不需要再传 build-arg。
- **`tr -d '[:space:]'` 比 `awk '{print $1}'` 干净**：VERSION 文件末尾的 `\n`、可能的多余空格、CR（Windows checkout）一次性全清掉。
- **`make help` 没显示 VERSION**：当前没加。如果未来想在 `make help` 里显示当前版本，加一行 `@echo "  version: $(VERSION)"`。本次没加是因为 help 是命令清单，VERSION 不是命令。
